# Рабочий процесс вайб-кодера

Как строить реальный рабочий день с Claude Code — от идеи до деплоя. Не теория, а конкретные шаги.

## Утро: планирование с AI

Не начинай кодить без ясной цели. 5 минут на планирование — экономят 2 часа путаницы.

```
Промпт для старта дня:
«У меня есть [продукт/проект]. Сегодня хочу сделать [функцию].
Разбей на задачи по 30-60 минут. Для каждой — что конкретно нужно сделать 
и какие могут быть подводные камни.»
```

Пример вывода:
```
Задача: добавить подписку Stars в бота

1. (30 мин) Создать таблицы в БД (users + purchases)
2. (45 мин) Написать хендлер /subscribe с InvoiceBuilder  
3. (20 мин) Хендлер pre_checkout_query + successful_payment
4. (30 мин) Middleware для проверки статуса подписки
5. (20 мин) Тест: купить через тестовые Stars

Подводные камни:
- Stars нельзя вернуть — проверь логику idempotency (charge_id UNIQUE)
- pre_checkout_query должен отвечать в течение 10 сек
```

## Работа с CLAUDE.md

CLAUDE.md — главный файл контекста. Держи его актуальным, он экономит токены и сокращает объяснения.

```markdown
# CLAUDE.md в корне проекта

## Контекст
Telegram-бот подписочного сервиса AI-ответов. aiogram 3 + aiosqlite + APScheduler.

## Структура
- bot.py — точка входа, роутеры
- db.py — все работы с БД  
- handlers/ — хендлеры по фичам
- middleware/ — авторизация, лимиты

## Паттерны
- Все DB-функции — async def + aiosqlite.connect(DB_PATH)
- Ошибки — логировать через logger.error(), не print()
- Новый хендлер — добавить в соответствующий файл handlers/

## Что НЕ делать
- Не использовать sync sqlite3 (только aiosqlite)
- Не хранить состояние в глобальных переменных
```

Чем точнее CLAUDE.md — тем меньше приходится объяснять в каждом промпте.

## Итерационный подход

Не пиши всё сразу — итерируй маленькими шагами:

```
1. Минимальная версия → работает
   «Сделай простой хендлер /subscribe который просто показывает цены»

2. Добавь логику → работает
   «Добавь отправку инвойса при нажатии на кнопку»

3. Добавь обработку ошибок → работает
   «Добавь обработку ошибки если Stars недостаточно»

4. Тест → исправь → деплой
```

Vs. «сделай сразу всё»: при ошибке на 5-м шаге непонятно что сломалось.

## Отладка с AI

```
Промпт для отладки (давай конкретику):
«Запускаю бота, при нажатии кнопки покупки получаю ошибку:

aiogram.exceptions.TelegramBadRequest: Bad Request: CURRENCY_TOTAL_AMOUNT_INVALID

Код хендлера:
[вставь 10-20 строк кода вокруг ошибки]

Что не так и как исправить?»
```

Никогда не пиши «у меня ошибка» без трейсбека и кода — это пустая трата токенов.

## Коммиты: часто и осмысленно

```bash
# Хорошая привычка: коммитить после каждой работающей итерации
git add -p  # добавлять по кускам, а не всё сразу
git commit -m "feat: add Stars payment handler"

# Если что-то сломалось:
git stash   # отложить изменения
git stash pop  # вернуть
```

Цель: никогда не терять работающую версию.

## Работа со сложными задачами

Когда задача большая или непонятная — сначала исследуй, потом кодируй:

```
Промпт для исследования:
«Хочу добавить в FastAPI WebSocket для real-time обновлений заказов.
Объясни архитектуру (не пиши код): как правильно структурировать,
какие есть подходы, что выбрать для моего случая
(1-100 активных пользователей, FastAPI + PostgreSQL).»
```

После понимания архитектуры — попроси код.

## Шаблон промпта для новой фичи

```
Контекст: [краткое описание проекта из CLAUDE.md]

Задача: [что нужно сделать]

Требования:
- [конкретное требование 1]
- [конкретное требование 2]

Технический стек: [что используешь]

Что уже есть: [существующий код который надо учесть]

Сделай минимальную рабочую версию без лишнего.
```

## Работа с legacy-кодом

Когда нужно разобраться в чужом или старом коде:

```
«Прочитай этот файл и объясни:
1. Что он делает в целом
2. Какие функции ключевые
3. Что может быть проблемой при изменении

[вставить код]»
```

Только после понимания — просить изменить.

## Когда AI галлюцинирует

Признаки:
- Использует несуществующий метод (`.fetch_one()` вместо `execute_fetchone()`)
- Ссылается на версию API которой нет
- Уверенно говорит что «так и работает», но не работает

Что делать:
1. Проверить документацию (docs.aiogram.dev, etc.)
2. Уточнить: «Ты уверен что этот метод существует в aiogram 3.x? Покажи откуда это.»
3. Попросить альтернативу: «Этот метод не существует. Какой правильный способ?»

## Сохранение контекста между сессиями

Claude Code не помнит прошлые сессии. Решение:

```markdown
# TODO.md (держи в корне проекта)

## Сегодня работаю над
- [ ] Добавить подписку Stars

## Следующие задачи
- [ ] Страница аналитики в боте
- [ ] Email-уведомления при оплате

## Решённые проблемы (на будущее)
- aiosqlite.Row не поддерживает индексацию после версии 0.19 — используй dict(row)
- В aiogram 3 pre_checkout_query нельзя игнорировать — обязательно ответить
```

В начале сессии: `/read TODO.md` или вставь в контекст.

## Типичный продуктивный день

```
09:00 — Планирование с AI (5 мин)
09:05 — Первая задача: минимальная версия
09:45 — Работает → коммит → следующая задача
11:00 — Обед / перерыв
14:00 — Вторая задача: более сложная логика
15:30 — Ошибка → отладка с AI → fix → коммит
17:00 — Тестирование → небольшие фиксы
17:30 — Обновить TODO.md → деплой (если готово)
```

Ключевой принцип: **маленькие рабочие итерации**, не большие монолитные куски.

---

::: info Производительность
Самая частая потеря времени — «строить в уме» вместо «проверять в реальности». Написал 20 строк → запустил. Если работает → идём дальше. Если нет → исправляем конкретную ошибку. Не планируй архитектуру на 3 часа вперёд до первого рабочего кода.
:::
