# Как работает интернет: HTTP, запросы, ответы

Когда ты пишешь код который обращается к API или создаёшь сервер — за этим стоит HTTP. Понять как это устроено значит понять почему всё работает именно так.

## Что происходит когда открываешь сайт

1. Ты вводишь `https://example.com` в браузере
2. Браузер спрашивает DNS: «Какой IP у example.com?» → получает `93.184.216.34`
3. Браузер устанавливает TCP-соединение с сервером на этом IP
4. Браузер отправляет HTTP-запрос: «Дай мне страницу /»
5. Сервер отвечает HTML-кодом
6. Браузер рисует страницу

Весь обмен данными — это HTTP-запросы и ответы.

## Анатомия HTTP-запроса

```
GET /api/users?page=1 HTTP/1.1
Host: api.example.com
Authorization: Bearer токен123
Content-Type: application/json
User-Agent: Python/requests
```

Части:
- **Метод** (`GET`) — что делаем
- **Путь** (`/api/users?page=1`) — к чему обращаемся
- **Заголовки** — метаданные (кто мы, что принимаем, авторизация)
- **Тело** — данные которые отправляем (для POST/PUT)

## HTTP-методы

| Метод | Значение | Пример |
|-------|----------|--------|
| `GET` | Получить данные | Загрузить список задач |
| `POST` | Создать что-то новое | Добавить задачу |
| `PUT` | Заменить полностью | Обновить всю задачу |
| `PATCH` | Частично обновить | Изменить только статус |
| `DELETE` | Удалить | Удалить задачу |

В браузере через адресную строку — только GET. POST и остальные — через JavaScript fetch или Python requests.

## Коды статусов

Сервер всегда возвращает числовой код:

```
200 OK             — всё хорошо, данные в ответе
201 Created        — создано (обычно после POST)
204 No Content     — успех, но ответа нет (обычно DELETE)
301 Moved Permanently  — редирект навсегда
302 Found          — временный редирект
400 Bad Request    — ты прислал что-то неправильное
401 Unauthorized   — нужна авторизация
403 Forbidden      — авторизован, но нет прав
404 Not Found      — не существует
429 Too Many Requests  — превышен rate limit
500 Internal Server Error  — сервер сломался
502 Bad Gateway    — ошибка прокси/nginx
503 Service Unavailable    — сервер недоступен
```

Запомни группы: 2xx — успех, 3xx — редирект, 4xx — твоя ошибка, 5xx — ошибка сервера.

## Заголовки запроса

```python
headers = {
    "Authorization": "Bearer токен",   # авторизация
    "Content-Type": "application/json", # что посылаю
    "Accept": "application/json",       # что хочу получить
    "User-Agent": "MyBot/1.0",         # кто я
}
```

**Content-Type** — формат тела запроса:
- `application/json` — JSON данные
- `multipart/form-data` — файлы
- `application/x-www-form-urlencoded` — HTML-форма

## JSON — главный формат данных

Большинство API работают через JSON. Это текст, структурированный как словарь:

```json
{
  "id": 42,
  "name": "Андрей",
  "tags": ["vibe-coder", "python"],
  "address": {
    "city": "Москва"
  },
  "active": true,
  "score": null
}
```

Python → JSON:
```python
import json
data = {"name": "Андрей", "age": 30}
json_str = json.dumps(data, ensure_ascii=False)
```

JSON → Python:
```python
data = json.loads('{"name": "Андрей"}')
print(data["name"])  # Андрей
```

Requests делает это автоматически:
```python
response = requests.get(url)
data = response.json()  # автоматически парсит JSON

requests.post(url, json=data)  # автоматически сериализует JSON
```

## HTTPS: шифрование

HTTP — передаёт данные открытым текстом. Любой посредник может прочитать.

HTTPS — то же самое, но с TLS-шифрованием. Данные видны только серверу и клиенту.

Как это работает:
1. Сервер имеет SSL-сертификат (от Let's Encrypt или другого CA)
2. При подключении сервер отдаёт сертификат
3. Браузер проверяет что сертификат настоящий
4. Устанавливается зашифрованное соединение

Зачем это важно:
- Токены авторизации не перехватываются
- Данные форм (пароли, платёжные данные) защищены
- Google лучше ранжирует HTTPS-сайты

## DNS — телефонная книга интернета

Доменное имя (`example.com`) → IP-адрес (`93.184.216.34`) — это задача DNS.

Типы DNS-записей:
- **A** — домен → IPv4 (основная запись)
- **AAAA** — домен → IPv6
- **CNAME** — псевдоним (`www` → `example.com`)
- **MX** — почтовый сервер домена
- **TXT** — текст (верификация, SPF для email)

Когда подключаешь домен к серверу — добавляешь A-запись с IP сервера. Изменения DNS распространяются до 48 часов (обычно быстрее).

Проверить DNS:
```bash
# Узнать IP домена
nslookup example.com

# Проверить все записи
dig example.com ANY

# Быстро: что отдаёт DNS прямо сейчас
host example.com
```

## Cookies и сессии

Cookie — небольшой файл который сервер просит браузер сохранить:

```
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict
```

При каждом следующем запросе к этому домену браузер автоматически присылает cookie обратно. Так сервер узнаёт «это тот же пользователь что логинился вчера».

- `HttpOnly` — JavaScript не может прочитать (защита от XSS)
- `Secure` — только через HTTPS
- `SameSite=Strict` — не отправляется при переходе с других сайтов (защита от CSRF)

## Как работает REST API

REST — это соглашение о том как строить API через HTTP:

```
GET    /users          — список пользователей
POST   /users          — создать пользователя
GET    /users/42       — пользователь с id=42
PUT    /users/42       — обновить пользователя 42
DELETE /users/42       — удалить пользователя 42
GET    /users/42/posts — посты пользователя 42
```

Принципы:
- URL описывает ресурс (существительное), метод — действие (глагол)
- Без состояния: каждый запрос содержит всё необходимое
- Стандартные статус-коды для разных ситуаций

Когда создаёшь API на FastAPI — ты следуешь этим соглашениям автоматически.
